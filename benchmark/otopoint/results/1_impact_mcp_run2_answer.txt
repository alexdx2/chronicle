## External Systems

### Redis
- **Pub/Sub channels** (via `OrderRedisService`):
  - `order:created:{merchantId}` — published on every new order; payload is the full serialized Order
  - `order:updated:{orderId}` — published on every state/payment change
  - `order:cancelled:{orderId}` — published on cancellation
- **Pattern subscriber** `order:*` in `OrderPubSubService` (`api/src/services/pubsub/order.pubsub.ts`) — if Order fields change, the deserialized payload shape breaks here
- **Printer/group context** in `PrinterRedisService` — stores `orderId` references inside user Redis contexts

### Socket.io (WebSocket)
Events emitted to rooms (via `SocketService`, `api/src/services/socket.service.ts`):

| Event | Triggered by | Consumers |
|-------|-------------|-----------|
| `order:purchased` | Order creation | Mobile app (customer confirms order placed) |
| `order:update` | State/payment changes | Mobile app (real-time status) |
| `orders:update` | Any order mutation | CRM merchant dashboard |
| `orders:new` | New order | CRM kitchen/waiter views |
| `order_item:state_changed` | Single item update | CRM station view |
| `order_item:bulk_state_changed` | Bulk item update | CRM station view |
| `voucher_update` / `lobby_voucher_update` / `merchant_voucher_update` | Cancellation (voucher release) | Mobile + CRM |
| `points_claimed` | Order closed + points awarded | Mobile app |

Both `mobile.gateway.ts` and `crm.gateway.ts` expose the Socket.io namespaces that receive these events.

### Stripe
- **`POST /stripe/webhook`** (`StripeWebhookController`) → `PaymentService.handlePaymentSucceeded()` → `OrderService.createOrderFromCart()`. If `Order.paymentStatus`, `Order.total`, or the Payment→Order relation changes, the fulfillment path breaks.
- **`PaymentService.createPaymentIntent()`** snapshots cart + voucher state before calling Stripe; the Payment record stores a cart snapshot that is later replayed into `createOrderFromCart` — tightly coupled to `CartItem` shape and `Order` fields.
- The `Payment` model has a 1-to-1 relation to `Order` (`order.payment`); renaming `orderId` on either side breaks this join.

### Centrifugo
- `api/src/services/centrifugo.service.ts` and `api/src/controllers/centrifugo.controller.ts` exist and are wired in `app.module.ts`. However, no direct Order-payload coupling was found in `centrifugo.service.ts` — Centrifugo appears to be used for the chat system, not order events directly. **Not a direct risk** from Order model changes, but worth verifying if chat messages carry embedded Order IDs (they do via `ChatMessage.orderId` FK in schema).

### PostgreSQL (Prisma cascades)
- `OrderItem` has `orderId FK → Order.id` with cascade delete (verify in schema before adding/removing fields)
- `Payment` has `orderId FK → Order.id` (1-to-1)
- `ChatMessage` has `orderId FK` for order placeholders
- `UserVoucher.orderId` — FK to Order for PENDING/USED vouchers

### `@otopoint/pricing-engine` (external package)
- `evaluatePromotions()` is called inside `OrderService` with Order/CartItem data. If Order item structure changes (e.g., `options`, `price`, `quantity` fields on `OrderItem`), this package's interface may break without a package update.
