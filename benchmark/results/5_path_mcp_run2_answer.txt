**spectators-api → tom-api** via `CALLS_SERVICE` (confidence: 0.95, hard derivation)

| Hop | Edge | Direction |

| spectators-api → jerry-api | `CALLS_SERVICE` (hard, 0.95) | spectators-api queries jerry-api for character info |
| arena-api → jerry-api | `CALLS_SERVICE` (hard, 0.95) | arena-api uses `JerryClient` to call `GET /jerry/status`, `GET /jerry/traps` |
| arena-api → tom-api | `CALLS_SERVICE` (hard, 0.95) | arena-api uses `TomClient` to call `GET /tom/status` |

Both spectators-api and tom-api are therefore downstream collaborators of **arena-api**, which acts as the battle orchestrator. Concretely:

1. `POST /arena/attack` triggers `ArenaService`
2. `ArenaService` calls `TomClient → tom-api` and `JerryClient → jerry-api`
3. After the battle, `BattleResultProducer` publishes to the **`battle-results`** Kafka topic
4. `BattleResultConsumer` in spectators-api consumes that topic to record results

This means **arena-api is the runtime bridge** — events originating from tom-api's state flow through arena-api and eventually land in spectators-api's stats.
